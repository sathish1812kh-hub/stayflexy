## 1. Architecture Options

We need to decide the deployment strategy for the Apollo Supergraph Gateway and the integration pattern for the `hotel-service` subgraph.

### Option A: Minimal Changes (BFF Gateway only)
Instead of fully adopting Federation immediately, we add an Apollo Server to the existing Node.js `api-gateway`. This server acts as a Backend-For-Frontend (BFF) and its resolvers simply make HTTP calls to the existing REST endpoints.
*   **Pros**: Fastest to implement. No changes required to `hotel-service` or any other microservice.
*   **Cons**: Does not actually use Apollo Federation. Hard to scale schema management as a single team must maintain the Gateway's GraphQL schema. Still suffers from internal network latency (N+1 HTTP calls between Gateway and Services).

### Option B: Clean Architecture (Standalone Apollo Router Edge)
Deploy the official **Apollo Router** (written in Rust) as a brand new Kubernetes deployment/service, operating alongside or replacing the current `api-gateway`. The `hotel-service` implements an `@apollo/subgraph` using Pothos code-first generation.
*   **Pros**: Enterprise-grade performance. Clear separation of concerns. Native support for Federation v2, telemetry, and query planning.
*   **Cons**: Requires managing a new Rust-based infrastructure component. High initial DevOps effort to set up schema composition pipelines (Apollo Studio or Rover CLI).

### Option C: Pragmatic Balance (Node.js Supergraph Gateway + Subgraph)
Upgrade the existing Express-based `api-gateway` to run `@apollo/gateway` alongside its existing REST proxy routes. Add `@apollo/subgraph` to the `hotel-service` (using Pothos) and compose them at runtime.
*   **Pros**: Reuses existing Node.js Gateway infrastructure (authentication, rate-limiting middlewares can be shared). Introduces true Federation allowing `hotel-service` to own its schema.
*   **Cons**: Node.js Gateway is slower at query planning than the Rust Apollo Router. Slight risk of gateway bloat if not carefully monitored.

### Trade-off Comparison

| Metric | Option A (BFF) | Option B (Rust Router) | Option C (Node Supergraph) |
|--------|----------------|------------------------|----------------------------|
| **Complexity** | Low | High | Medium |
| **Maintainability** | Low | High | High |
| **Effort** | Low | High | Medium |
| **Risk** | Low | Medium | Medium |

## 2. API Design (Target State)

### Supergraph (Apollo Router)
The new Gateway will be the official Rust-based Apollo Router. It will expose a single `/graphql` endpoint to the public frontend.

### Subgraph (`hotel-service`)
The `hotel-service` will expose an internal `/graphql` endpoint that serves an Apollo Subgraph schema, capable of being composed by the Router. It will respond to the `_service` and `_entities` queries required by Federation v2.

**Example `hotel-service` Schema (Auto-generated by Pothos):**
```graphql
extend schema @link(url: "https://specs.apollo.dev/federation/v2.0", import: ["@key"])

type Hotel @key(fields: "id") {
  id: ID!
  name: String!
  location: String!
  status: String!
}

type Query {
  hotel(id: ID!): Hotel
  hotels: [Hotel!]!
}
```

## 3. Data Model & Database Changes

No database migrations or Prisma schema changes are required for this phase. The GraphQL implementation will wrap existing `PrismaHotel` queries. 

However, to avoid N+1 issues when fetching nested relations (e.g., `RoomTypes` for a `Hotel`), the application layer must introduce **DataLoader** instances that batch Prisma queries (e.g., `findMany({ where: { hotelId: { in: keys } } })`).

## 4. Component Design

### 4.1 Apollo Router (New Component)
*   **Infrastructure:** A new Kubernetes deployment running the `ghcr.io/apollographql/router` image.
*   **Configuration:** `router.yaml` containing CORS policies, query complexity limits, and telemetry setup.
*   **Schema Composition:** Rover CLI will be used in CI/CD to compose the `supergraph.graphql` file from the subgraphs and provide it to the Router.

### 4.2 `hotel-service` GraphQL Module
*   **Server:** `@apollo/server` running alongside the existing Express app (e.g., via `expressMiddleware`).
*   **Schema Builder:** `@pothos/core` and `@pothos/plugin-federation` to programmatically build the GraphQL schema based on the existing TypeScript entities.
*   **Context & DataLoaders:** A GraphQL context object containing authenticated user data and DataLoader instances for batched database fetching.

## 5. Session Guide (Implementation Steps)

### Step 1: Infrastructure Setup (DevOps)
1. Add a Kubernetes deployment configuration for the Apollo Router (`infrastructure/kubernetes/services/apollo-router/deployment.yaml`).
2. Add the Router service configuration (`infrastructure/kubernetes/services/apollo-router/service.yaml`).
3. Set up the Rover CLI tool in the CI/CD pipeline to pull subgraph schemas and build the `supergraph.graphql`.

### Step 2: `hotel-service` Subgraph Initialization
1. Install dependencies: `@apollo/server`, `@apollo/subgraph`, `graphql`, `@pothos/core`, `@pothos/plugin-federation`, `dataloader`.
2. Initialize Pothos schema builder (`src/modules/graphql/builder.ts`).
3. Define the `Hotel` object type and Queries (`src/modules/graphql/schema/hotel.ts`).
4. Implement DataLoader for nested hotel relationships (`src/modules/graphql/dataloaders/index.ts`).
5. Expose the `/graphql` route using `expressMiddleware` in `src/main.ts`.

### Step 3: Gateway Configuration
1. Configure Apollo Router (`router.yaml`) to set up query complexity limits and disable introspection in production.
2. Provide the composed schema to the Router.

### Step 4: Validation & Testing
1. Execute queries against the Router to ensure the Supergraph successfully fetches data from the `hotel-service` Subgraph.
2. Analyze Prisma logs to confirm that DataLoader is successfully batching queries and preventing N+1 problems.
